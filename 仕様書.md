# Claude Code Remote Handoff CLI 仕様メモ

## 概要

このツールは、Claude Code CLI から現在の作業をリモートサーバーへ引き継ぎ、サーバー側で Claude Code セッションを継続実行できるようにする仕組みを提供する。Claude Code ではカスタム slash command を Markdown ベースで追加できるため、`/upload`、`/attach`、`/pull` のような操作をひとまとまりのワークフローとして提供できる。[cite:10][cite:87]

セッション継続には Claude Code の Remote Control を利用し、既存セッションや server mode、`worktree` ベースの分離運用が可能であることを前提にする。[cite:24][cite:26][cite:62]

## 目的

- ローカルの作業状態を必要最小限のファイルだけサーバーへ送る。
- サーバー側で Claude Code を起動し、作業を継続する。
- 進行状況をローカル CLI にストリーミング表示する。
- 必要に応じて Remote Control リンクを表示する。
- 後から `/attach` で再接続し、`/pull` で成果物を戻せるようにする。

## コマンド構成

| コマンド | 役割 |
|---|---|
| `/upload` | 現在のワークスペースをサーバーへ送り、リモート作業を開始する。 |
| `/attach [instance]` | 既存インスタンスに再接続し、ライブ出力または最新出力を見る。 |
| `/pull [instance]` | リモートインスタンスの成果物や現在状態をローカルへ戻す。 |

CLI の命名は責務を明確に分け、動詞中心で統一した方が理解しやすい。[cite:98][cite:110][cite:115]

## サーバー構成方針

サーバーは常時重い処理を抱えず、軽量な常駐デーモンだけを待機させ、実作業時にのみ Claude Code セッションを起動する構成とする。systemd socket activation を使えば、接続が来たときだけサービス本体を起動でき、常時リソース消費を抑えられる。[cite:45][cite:69][cite:78]

配布対象は Ubuntu、Arch、Docker を想定し、ネイティブ Linux では systemd service / socket、Docker では単一常駐プロセスを基本とする。[cite:45][cite:48]

## 認証とアクセス制御

アクセスは専用ユーザーと公開鍵認証を基本とする。OpenSSH 系では `authorized_keys` による公開鍵認証や接続元制限が一般的であり、`from=` オプションで接続元アドレス制限を追加できる。[cite:54][cite:55][cite:64]

初期方針は次のとおり。

- 専用ユーザーでのみ実行する。
- パスワード認証は使わない。
- 公開鍵認証を必須にする。
- オプションで鍵ごとに IP 制限を付ける。
- 将来的に鍵ごとの権限制御や forced command 相当の制限を検討する。[cite:74][cite:77]

## リモート作業ディレクトリ

同一リポジトリに対して複数の作業を平行で扱えるよう、Git worktree ベースの分離を基本にする。`git worktree` は複数の working tree を同一リポジトリに紐づけて管理できるため、タスクごとの分離と再接続に向いている。[cite:62][cite:89]

同じリポジトリの既存インスタンスが見つかった場合は、単純な上書き確認ではなく、次の3択を提示する。

- 既存インスタンスを再利用する。
- 新しいインスタンスを作る。
- 中止する。

## `/upload` のファイル選定方針

ファイル選定は `git + Claude 選定` の二段構えにする。まず Git 由来で候補を集め、その候補だけを Claude に見せて「継続に必要なもの」を選ばせる方が、探索範囲を絞れて扱いやすい。[cite:62][cite:89]

候補抽出の基本は次のとおり。

- 変更済みファイル。
- 未追跡ファイル。
- 依存解決やビルドに必要な設定ファイル。
- README など文脈把握に有効なファイル。
- 秘密情報や明らかな不要物は除外する。

## アップロードサイズ制限

初期仕様ではデフォルトのアップロード上限を 200MB にする。`--max-upload-size=0` を指定した場合は無制限とするが、サイズ警告は表示する。[cite:110][cite:115]

上限超過時は一律失敗ではなく、次の分岐を出す。

- Continue once
- Exclude large files and retry
- Re-run selection with size limit
- Cancel

確認プロンプトは、何が起きるかが分かる行動語ベースで書いた方が誤操作を減らしやすい。[cite:113][cite:116]

## `/attach` 対話フロー案

引数ありの場合は対象インスタンスへ直接つなぐ。引数なしの場合は recent instances から選ばせる。[cite:103][cite:105]

```text
Attaching to "login-fix-3"...
Press Ctrl+C to stop watching.
```

```text
Recent remote instances:
1) login-fix-3    running
2) auth-cleanup   idle
3) test-run-4     completed
4) Cancel

Choose an instance:
> 
```

対象が completed の場合は、ライブではなく最新出力を表示する。

```text
Instance "test-run-4" is not running.
Showing the latest available output instead.
Press Ctrl+C to close.
```

## `/pull` 対話フロー案

```text
Preparing to pull "login-fix-3"...
Checking remote workspace...
```

running 中のインスタンスに対しては、現在状態を pull するか attach に切り替えるかを選ばせる。

```text
Instance "login-fix-3" is still running.

Choose how to continue:
1) Pull current state
2) Attach instead
3) Cancel
> 
```

ローカル復元先が競合する場合は次を出す。

```text
A local directory for this repo already exists.

Choose how to continue:
1) Restore into a new directory
2) Overwrite the existing restore directory
3) Cancel
> 
```

成功時は復元先を示す。

```text
Pull complete.
- Instance: login-fix-3
- Restored to: ./remote/login-fix-3

Next actions:
- Review output: /attach login-fix-3
```

## 将来拡張

他の PC へ接続情報を送る機能を将来的に追加したい場合、秘密鍵そのものを共有するのではなく、接続先・インスタンスID・サーバー fingerprint・有効期限などを持つ session ticket 形式にすると拡張しやすい。鍵認証とコマンド制限を分離できるため、共有機能の安全性を上げやすい。[cite:74][cite:77]

例:

```go
type SessionTicket struct {
    Version     int
    InstanceID  string
    ServerHost  string
    ServerPort  int
    User        string
    HostKeyFP   string
    RemoteURL   string
    ExpiresAt   time.Time
}
```

 -> それでいいと思う
4. `/pull` を tar.gz 復元から始めるか、差分同期まで最初から入れるか。
　-> git pullみたいな感じで実装したい
